## 一、项目概述

Octopus 是一套**自主研发的企业级 AI Agent 智能协作平台**，面向企业内网环境，提供安全可控的多智能体协作能力。平台采用**双网关分离架构**，在原生 AI 引擎之上构建了完整的企业级能力层，实现了多用户隔离、权限管控、审计追踪、即时通讯集成等核心功能。

### 核心定位

| 维度 | 说明 |
|------|------|
| **目标场景** | 企业内网部署，数据不出域，安全合规 |
| **核心能力** | 多 Agent 智能协作、知识记忆、技能扩展、工具集成 |
| **用户群体** | 企业全员（通过飞书/微信/Web 三端接入） |
| **技术特色** | 单进程架构、Docker 沙箱隔离、插件化扩展 |

## 二、安全性：企业级纵深防御体系

> **设计理念：安全靠架构硬执行，不靠文字指令软约束。**

Octopus 构建了**六层纵深防御体系**，从认证、隔离、权限、审计到熔断，形成完整的安全闭环。

### 2.1 多维度沙箱隔离

**解决的问题：** AI Agent 需要执行代码、操作文件，如何防止越权访问和数据泄露？

平台采用 **Docker 容器沙箱** 实现三维隔离：

| 隔离维度 | 实现方式 | 防护效果 |
|---------|---------|---------|
| **用户隔离** | 容器内 uid=2000（非宿主机用户），防止 bind mount 权限穿透 | 即使容器逃逸也无法读取宿主机敏感文件 |
| **网络隔离** | 独立 Docker 网络 172.30.0.0/16 + iptables 规则 | 容器可访问内网服务，但**公网完全封锁** |
| **文件隔离** | 仅挂载用户专属 workspace 目录，`workspaceOnly=true` 强制限制 | 每个用户/Agent 只能读写自己的工作空间 |

**网络隔离规则详解：**
- 允许访问：企业内网（192.168.0.0/16、10.0.0.0/8）
- 完全封锁：互联网出站流量
- 端口封锁：MySQL(3306)、Redis(6379)、网关(18790/19791) 等敏感端口
- 效果：即使 Agent 被恶意指令注入，也无法将数据发送到外部

### 2.2 用户命名空间隔离

**解决的问题：** 多用户共享同一平台，如何确保数据互不可见？

平台为每个用户的每个 Agent 分配唯一命名空间：

```
命名空间格式：ent_{用户ID}_{Agent名称}
会话密钥：agent:ent_{用户ID}_{Agent名称}:session:{会话ID}
工作目录：data/users/{用户ID}/workspace/
```

**隔离覆盖范围：**
- Agent 配置：用户只能管理自己的 Agent
- 会话数据：用户只能查看自己的对话历史
- 文件存储：每个用户独立工作空间，物理隔离
- AI 记忆：每个 Agent 的记忆按 agentAccess 白名单访问（详见记忆系统章节）
- 所有权校验：使用 `startsWith` 前缀匹配（防止构造伪造 ID 绕过）

### 2.3 多层权限管控

**解决的问题：** 不同角色的 Agent 需要不同的工具能力，如何精细化控制？

平台实现了**三维度白名单权限模型**：

| 权限维度 | 控制对象 | 实现机制 |
|---------|---------|---------|
| **工具权限** (toolsFilter) | 文件读写、命令执行 | 引擎 `tools.allow` 白名单 |
| **技能权限** (skillsFilter) | 可调用的企业技能 | `tools.deny: ["run_skill"]` + 提示词注入控制 |
| **MCP权限** (mcpFilter) | 外部工具/数据库连接 | 双层检查：提示词隐藏 + 执行层验证 |

**权限执行为"双保险"设计：**
1. **软限制（提示词层）：** SystemPromptBuilder 按白名单过滤，未授权的工具不在提示词中出现，AI 不知道其存在
2. **硬限制（执行层）：** 即使 AI 被注入尝试调用未授权工具，执行层会再次校验权限并拒绝

### 2.4 企业级审计系统

**解决的问题：** 企业合规要求所有 AI 操作可追溯、可审计、防篡改。

平台实现了**双写审计日志 + HMAC 签名链**：

**双写策略：**
- **数据库写入**：MySQL 实时存储，支持即时查询和合规检查
- **文件写入**：JSONL 格式按天归档，支持离线分析和长期保存

**防篡改机制（HMAC-SHA256 签名链）：**
```
每条日志 = 内容 + 前一条的哈希 + 当前哈希
```
- 形成链式签名：每条记录包含前一条的哈希值
- 删除检测：如果删除中间任何一条，后续记录的哈希链将断裂
- 篡改检测：修改任何一条记录，其哈希值将与后续记录不匹配

**审计覆盖 20 个关键操作点：**
- Agent 创建/编辑/删除
- 对话消息发送/接收
- 工具调用（记录参数键名，不记录敏感值）
- 文件上传/下载
- IM 绑定/切换
- 权限变更

### 2.5 认证授权与安全防护

| 能力 | 实现 |
|------|------|
| **企业认证** | JWT + LDAP/AD 目录集成（支持 Mock 开发模式） |
| **Token 安全** | 访问令牌 2h 过期 + 刷新令牌 7d，DB 校验用户有效性 |
| **暴力破解防护** | 同一 IP+用户 1 分钟内失败超 10 次触发告警 |
| **已删用户拦截** | JWT 有效但用户已删除 → 立即拒绝（从 DB 校验最新状态） |
| **MCP 环境变量隔离** | 子进程不继承 process.env 全量，仅传递白名单变量 |

### 2.6 AI 行为熔断机制

**解决的问题：** AI Agent 可能陷入工具调用死循环，无限消耗资源。

平台实现了**三层熔断防护**：

| 层级 | 阈值 | 动作 |
|------|------|------|
| 警告层 | 同一工具连续调用 8 次 | 向 AI 发送警告信息 |
| 拒绝层 | 连续 15 次 | 工具调用被直接拒绝 |
| 终止层 | 全局累计 25 次 | 整个对话运行终止 |

此外，MCP 工具额外实现了**连续失败熔断**：同一工具连续失败 3 次后自动熔断 5 分钟，防止错误扩散。

## 三、创新性：核心功能详解

### 3.1 智能记忆系统

> **让 AI 拥有"长期记忆"，越用越聪明。**

**解决的问题：** 传统 AI 对话是"无状态"的，每次对话都从零开始。企业场景中，AI 需要记住用户偏好、项目背景、历史决策，才能提供真正有价值的持续服务。

**技术实现：**

平台集成 **LanceDB 向量数据库**，实现了完整的长期记忆能力：

```
用户对话 → 自动提取关键信息 → 向量化存储（Jina Embedding）
                                          ↓
后续对话 → 语义检索相关记忆 → 注入上下文 → AI 给出更精准的回答
```

**核心特性：**

| 特性 | 说明 | 价值 |
|------|------|------|
| **混合检索** | 向量语义匹配(70%) + BM25 关键词匹配(30%) | 兼顾语义理解和精确匹配 |
| **时间衰减** | 14 天半衰期，60 天完全衰减 | 自动淡忘过期信息，保持记忆新鲜 |
| **智能提取** | 自动识别对话中的关键事实和决策 | 无需用户手动标记，零成本积累知识 |
| **交叉重排序** | Jina Reranker v3 对候选结果精排 | 确保最相关的记忆被优先召回 |
| **角色级隔离** | 主 Agent 可读所有下属记忆，专业 Agent 仅读自己+主 Agent | 安全共享，防止跨角色泄露 |

**业务价值举例：**
- 用户第一次告诉 AI "我们的报表用 ECharts 5 生成"，后续所有报表任务 AI 自动使用 ECharts
- 项目经理与 AI 讨论了数据库设计决策，半年后新同事接手，AI 仍记得设计背景和约束
- 数据分析师的专属 Agent 积累了部门的数据字典和分析模板，成为团队知识库

### 3.2 MCP 工具集成平台

> **让 AI 能操作企业内部系统，从"对话"升级为"执行"。**

**解决的问题：** AI 仅能对话远远不够，企业需要 AI 能直接操作数据库查询、发送邮件、调用内部 API——但又必须安全可控。

**什么是 MCP（Model Context Protocol）：**

MCP 是模型上下文协议，是连接 AI 与外部工具的标准化桥梁。通过 MCP，AI 可以安全地调用企业内部的数据库、API、文件系统等各类工具。

**平台的 MCP 实现创新：**

```
管理员定义 MCP Server → 平台自动发现工具 → 按权限分配给 Agent → AI 安全调用
```

**三种运行模式：**

| 模式 | 适用场景 | 隔离级别 | 说明 |
|------|---------|---------|------|
| **宿主机直连** | 企业受信基础设施 | 环境变量白名单 | 企业 MCP，管理员配置 |
| **Docker 沙箱** | 用户自定义工具 | 完整容器隔离 | 个人 MCP，不信任代码 |
| **HTTP 远程** | 第三方服务 | 网络级隔离 | 远程 MCP Server |

**企业级增强能力：**
- **权限白名单**：每个 Agent 只能访问管理员授权的 MCP 工具和数据库连接
- **环境变量安全**：MCP 子进程不继承宿主机完整环境变量，仅注入白名单变量
- **数据库连接隔离**：每个用户的数据库凭证独立加密存储（AES-256-GCM），按需注入
- **工具失败熔断**：连续失败 3 次自动熔断，防止 AI 无限重试
- **工具缓存**：两级缓存（磁盘 + 内存），减少重复发现开销

**业务价值举例：**
- 数据分析师对 AI 说"帮我查上月销售额前 10 的产品"，AI 直接查询数据库返回结果
- 运维人员对 AI 说"发一封邮件通知团队系统维护时间"，AI 通过邮件工具直接发送
- 财务人员只能通过 AI 访问财务数据库，无法访问其他部门数据

### 3.3 企业技能系统 (Skill)

> **将复杂的专业流程封装为"一键可用"的 AI 技能。**

**解决的问题：** 企业中有大量标准化流程（报表生成、数据分析、PPT 制作等），靠 AI 每次从头理解效率低、质量不稳定。技能系统将这些流程标准化、可复用。

**什么是 Skill：**

Skill（技能）是预定义的专业能力包，包含执行脚本、示例数据、操作指南。AI 通过调用技能，可以稳定、高效地完成复杂任务。

**技能架构：**

```
技能包结构：
├── SKILL.md          # 操作指南（AI 阅读后知道怎么用）
├── scripts/          # 执行脚本（Python/Node.js）
│   ├── main.py       # 主入口
│   └── utils.py      # 辅助工具
├── examples/         # 示例数据（帮助 AI 理解输入输出格式）
└── templates/        # 模板文件（报表模板、PPT 模板等）
```

**技能管理模式：**

| 类型 | 管理方 | 可见范围 | 隔离方式 |
|------|-------|---------|---------|
| **企业技能** | 管理员统一部署 | 按 Agent 权限白名单分配 | 引擎原生发现 |
| **个人技能** | 用户自行上传 | 仅上传者可用 | Docker 沙箱执行 |

**执行安全保障：**
- 所有技能在 Docker 容器中执行，与宿主机隔离
- 共享 Python 虚拟环境（预装 numpy/pandas/matplotlib 等常用库），无需每次安装
- 脚本只能读写用户 workspace，无法访问其他目录
- 执行结果自动输出到 `outputs/` 目录，用户可直接下载

**已部署的技能示例：**
- **claude-code-dispatch**：将编码任务派发给 Claude Code 后台执行，支持异步通知
- **clawteam-research**：多智能体协作的调研/头脑风暴/发言稿生成

**业务价值举例：**
- 用户说"帮我分析这份销售数据"，AI 调用数据分析技能，自动生成 ECharts 可视化报表
- 用户说"把这份报告做成 PPT"，AI 调用 PPT 生成技能，直接输出成品文件
- 管理员部署新技能后，全企业 Agent 立即可用，无需用户配置

### 3.4 多 Agent 智能协作

> **主 Agent 统筹调度，专业 Agent 各司其职，团队化 AI 协作。**

**解决的问题：** 单一 AI 难以精通所有领域。企业需要不同专业方向的 AI 助手（数据分析、文案撰写、代码开发等），并能协同工作。

**协作模型：**

```
用户 ←→ 主 Agent（通用助手，统筹协调）
              │
              ├──委派──→ 数据分析 Agent（专精数据处理和可视化）
              ├──委派──→ 文案写作 Agent（专精内容创作和排版）
              └──委派──→ 技术开发 Agent（专精代码编写和调试）
```

**主 Agent vs 专业 Agent：**

| 能力 | 主 Agent | 专业 Agent |
|------|---------|-----------|
| 任务委派 | 可以派任务给专业 Agent | 不能再委派（防止递归） |
| 工具权限 | 按管理员配置 | 按管理员配置（独立） |
| 记忆访问 | 可读所有下属 Agent 记忆 | 仅读自己 + 主 Agent 记忆 |
| 工作空间 | 用户根目录 | 独立 Agent 目录 |
| 人设定制 | 通用助手 SOUL.md | 专业领域 SOUL.md |

**Agent 人设系统（IDENTITY.md + SOUL.md）：**
- **IDENTITY.md**：定义 Agent 的名称、头像、性格特征（creature + vibe）
- **SOUL.md**：定义 Agent 的核心指令、行为规范、领域专长
- 引擎根据这些文件渲染 Agent 人设，确保 AI 回复风格一致

**委派安全控制：**
- 通过 `allowAgents` 白名单控制可委派的目标 Agent
- 专业 Agent 的 `tools.allow` 中不包含 `sessions_spawn`，从引擎层面禁止嵌套委派
- 委派结果自动回传主 Agent，前端轻量轮询检查完成状态

**业务价值举例：**
- 用户对主 Agent 说"帮我分析上季度数据并写一份汇报"，主 Agent 自动将数据分析委派给数据 Agent、将报告撰写委派给文案 Agent，最终整合输出
- 每个部门可定制专属 Agent，积累部门知识，新人入职即可获得 AI 辅助

### 3.5 心跳巡检监控系统

> **AI Agent 自主巡检，主动发现问题并告警。**

**解决的问题：** 传统监控靠人工配置规则，难以覆盖所有场景。心跳系统让 AI Agent 定时自主检查工作空间状态，用自然语言理解力发现异常。

**工作原理：**

```
管理员配置巡检任务（Web UI）
         ↓
平台写入 HEARTBEAT.md（巡检指令）
         ↓
原生 Cron 定时触发 Agent 执行巡检
         ↓
Agent 自主检查 → 结果包含 "HEARTBEAT_OK" 则正常
         ↓
异常时通过 IM（飞书/微信）推送告警
```

**HEARTBEAT.md 示例：**
```markdown
## 巡检任务
- 检查 outputs/ 目录是否有未处理的文件
- 如有则打包成 ZIP 并在记忆中记录
- 检查最近 24h 有无异常日志
- 如果一切正常，回复 HEARTBEAT_OK
```

**核心特点：**
- 巡检任务用自然语言描述，无需编写监控脚本
- Agent 拥有完整的工具能力（读文件、执行命令、查数据库），可进行深度检查
- 不仅能"发现问题"，还能"处理问题"（如自动清理、自动归档）
- 巡检结果通过 IM 渠道实时推送，确保及时响应

**业务价值举例：**
- 每天 9 点自动检查数据文件是否按时到位，缺失则告警
- 定期检查工作空间磁盘使用，超限自动清理临时文件
- 自动扫描日志中的异常模式，发现后立即推送飞书通知

### 3.6 多渠道 IM 集成

> **飞书、微信、Web 三端统一接入，随时随地与 AI 协作。**

**解决的问题：** 员工日常工作在飞书/微信中，切换到 Web 界面使用 AI 增加操作成本。IM 集成让 AI 融入日常工作流。

**架构设计（适配器模式）：**

```
IMService（统一入口）
    ├── FeishuAdapter（飞书）  ← Webhook 接入
    ├── WeixinManager（微信）  ← iLink Bot API，支持多账号
    └── IMRouter（消息路由）
         ├── /bind     → 账号绑定
         ├── /agent    → 切换 Agent
         ├── /cancel   → 取消执行中的任务
         ├── /status   → 查看绑定状态
         └── 普通消息   → 路由到当前激活的 Agent
```

**核心能力：**

| 能力 | 说明 |
|------|------|
| **账号绑定** | `/bind username password`，验证后密码消息立即删除（安全） |
| **Agent 切换** | `/agent 数据分析师` 在 IM 中自由切换专业 Agent |
| **文件传输** | Agent 生成的文件（报表、图片等）自动发送到 IM |
| **超时保护** | 30 分钟执行兜底，超时自动取消并通知 |
| **心跳推送** | 巡检结果通过 IM 实时推送 |
| **多账号** | 微信支持多机器人并联，扩展服务容量 |

**业务价值举例：**
- 在飞书群里 @AI 说"帮我查一下今天的订单量"，AI 直接查库返回结果
- 在微信中切换到数据分析 Agent，发送 Excel 文件，AI 自动分析并返回图表
- 心跳巡检发现异常，自动推送飞书通知到运维群

## 四、前瞻性：架构设计与未来演进

### 4.1 插件化架构——无侵入扩展

平台采用**插件化架构**，所有企业级能力通过插件实现，不修改核心引擎：

| 插件 | 功能 | 说明 |
|------|------|------|
| **enterprise-audit** | 审计日志 | 20 个 hook 点，双写 DB + JSONL |
| **enterprise-mcp** | 工具集成 | 从 DB 动态发现 MCP Server，注册为原生工具 |
| **enterprise-email** | 邮件发送 | Nodemailer + SMTP，支持 AI 直接发邮件 |
| **memory-lancedb-pro** | 长期记忆 | 向量数据库，混合检索 + 角色隔离 |

**前瞻价值：**
- 新增企业能力只需开发新插件，不影响核心稳定性
- 插件可独立升级、启用/禁用
- 符合开闭原则：对扩展开放，对修改关闭

### 4.2 配置热加载——零停机更新

平台实现了**智能配置热加载**机制：

- `config.set` RPC 按路径 diff，自动判断变更影响范围
- Agent 配置变更 → 热加载（不重启引擎）
- 插件配置变更 → 仅在必要时重启
- 乐观并发控制（baseHash），防止配置覆盖冲突

**前瞻价值：** 生产环境可在线调整 Agent 配置、模型参数、权限策略，无需停机。

### 4.3 内网优先——数据主权

平台设计之初即面向**企业内网部署**：

| 设计决策 | 说明 |
|---------|------|
| **无外部依赖** | 除大模型 API 外，所有组件内网自部署 |
| **数据不出域** | 对话数据、审计日志、用户文件全部存储在企业内部 |
| **模型可替换** | 支持 DeepSeek、Claude、OpenAI Codex 等多种模型，通过内网代理接入 |
| **LDAP/AD 集成** | 无缝对接企业现有用户目录 |

**前瞻价值：** 满足数据安全和信息合规要求，适用于金融、政务、国企等对数据主权有严格要求的行业。

### 4.4 多模型适配——大模型无关

平台支持**多大模型并行接入**，不绑定特定厂商：

- DeepSeek（内网代理）：日常对话、文档处理
- Claude（antigravity 反代）：复杂推理、代码生成
- OpenAI Codex：编程任务
- MiniMax：特定场景

**前瞻价值：** 大模型技术快速迭代，平台可随时切换/新增最优模型，保护技术投资。

### 4.5 单进程架构——极致效率

平台创新性地采用**单进程内嵌架构**：

```
传统方案：Enterprise Gateway ←WebSocket RPC→ Native Engine（两个进程）
Octopus：Enterprise Gateway ←函数调用→ Native Engine（单进程内嵌）
```

**优势：**
- 零网络开销：进程内函数调用替代 TCP 通信
- 配置原子性：单进程内 Read-Modify-Write，无分布式一致性问题
- 运维简单：一键启动，无需管理多个服务间的依赖
